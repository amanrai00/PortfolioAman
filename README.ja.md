# ポートフォリオサイト

フロントエンドのスキルを見せるために、ゼロから作った個人ポートフォリオサイトです。テンプレートやテーマは一切使わず、設計からコードまですべて自分で考えて構築しました。

> 🔗 **公開中:** [obliviousaman.netlify.app](https://obliviousaman.netlify.app/)

---

## 使用技術

| 技術 | 用途 |
|------|------|
| Vue 3（Composition API） | `<script setup>` 構文をベースにしたコアフレームワーク |
| Vite | ビルドツール・開発サーバー |
| Tailwind CSS | ユーティリティファーストのスタイリング（カスタム設定あり） |
| GSAP + ScrollTrigger | スクロール連動アニメーション・ページ遷移 |
| Vue I18n | 英語 ↔ 日本語のリアルタイム切り替え |
| Lottie | 軽量なモーショングラフィックス |
| Netlify | ホスティング・デプロイ |

---

## 主な機能

- **バイリンガル対応（EN/JP）** — 単なるテキスト翻訳ではなく、言語ごとにレイアウトの余白やフォントサイズを調整。日本語の文字幅に合わせた表示を実現
- **スクロール連動アニメーション** — GSAP + ScrollTrigger で各セクションが表示時にアニメーション。画面外のセクションは遅延読み込みで無駄な初期化を防止
- **カスタムモバイルナビ** — ライブラリ不使用。円形に展開するハンバーガーメニューを独自実装
- **レスポンシブ対応** — スマホ・タブレット・デスクトップで一貫した動作を確認済み
- **セクションベース設計** — 各パートが独立したVueコンポーネント。保守性が高く、拡張しやすい構成

---

## 制作の流れ

1. **ワイヤーフレーム** — 紙にラフレイアウトを描き、必要なセクションとスクロールの流れを設計
2. **コンポーネント設計** — 早い段階でVueコンポーネントに分割。各セクションにアニメーションロジックを内包させる構成に
3. **アニメーション先行、コンテンツ後** — GSAPのスクロールトリガーを先に実装してからテキストを調整。静的なレイアウトに後からアニメーションを無理に乗せるのではなく、動きに合わせてタイミングを最適化
4. **多言語対応** — ベースの構造が固まった後にVue I18nを導入。日本語と英語では同じフォントサイズでもテキストの挙動が異なるため、余白やレイアウトを再調整
5. **モバイル最適化** — 想定以上に時間がかかった部分。ハンバーガーメニューだけで3回作り直し、円形展開の動きが自然になるまで繰り返した
6. **テスト・デプロイ** — Chrome・Safari・Firefox・モバイルブラウザで検証。GitHubからNetlifyへの自動デプロイを設定

## 学んだこと

### Vue 3 Composition API
- `<script setup>` とコンポーザブルを使ったコンポーネント設計（Options APIからの移行）
- 複数セクションにまたがる `ref` / `reactive` でのリアクティブな状態管理
- アニメーションとi18nが同時に絡む場面でのロジックの整理

### アニメーション・GSAP
- ScrollTriggerのpin・scrub機能を使った、カクつきのないスクロール連動アニメーション
- `duration` や `ease` の微調整がサイト全体の印象を大きく左右すること
- アニメーションインスタンスの遅延読み込みで、重いページでもブラウザの負荷を軽減

### 国際化（i18n）
- Vue I18nによる英語・日本語のランタイム切り替え
- ローカライズは単なるテキスト置換ではないことを実感。日本語は文字幅が広く、行間の調整も必要で、英語で問題なかったレイアウトが日本語で崩れるケースが多々あった
- サイトの拡張に伴う翻訳ファイルの管理と同期

### CSS・レスポンシブデザイン
- Tailwindのユーティリティクラスは効率的だが、独自のカラーパレットやブレークポイントにはカスタム設定が必要
- モバイルファーストで作ったおかげで、後からメディアクエリを書き直す手間を省けた
- スクロールアニメーションと固定ナビバーが重なると、z-indexの管理がすぐ複雑になる

### 問題解決
- GSAP × Vueのリアクティビティの競合 — DOMの準備前にアニメーションが発火する問題を `nextTick` とライフサイクルフックで解決
- IntersectionObserver → ScrollTrigger への切り替え — アニメーション進行の細かい制御が必要になったため移行
- 円形メニューのアニメーション — チュートリアルでは見つからなかった `clip-path` の手動計算が必要だった

## 全体を通しての成長

このプロジェクトで「Vueを知っている」から「Vueで実際にものを作れる」に変わりました。それまではチュートリアルに沿った小さな演習がほとんどで、すべての判断を自分でしたのはこれが初めてです。フレームワークの選定、アニメーションライブラリの選択、フォルダ構成、デプロイまで全部自分で決めました。

一番大きな変化は、チュートリアルなしでデバッグできるようになったことです。GSAPのアニメーションとVueのリアクティビティが競合したとき、公式ドキュメントを読み、実験し、自分で解決しました。「学ぶ」と「作る」の間にある壁を、このプロジェクトで超えました。

2言語で作ったことで、自分以外のユーザーのことを考えるようにもなりました。このサイトは日本語で「動く」だけじゃなく、日本語で「自然に感じる」ことを目指しています。その違いは、実際にやってみないとわからないものでした。

## 改善できるポイント

- **アクセシビリティ** — ARIAラベルの追加、キーボード操作の改善、スクリーンリーダーでのテスト
- **パフォーマンス最適化** — ルートごとのコード分割、画像の遅延読み込みとWebP化
- **ダーク/ライトモード** — 現在は1テーマのみ。トグル機能を追加してユーザーの好みに対応
- **CMS連携** — コンテンツがハードコードのため更新が手間。Contentfulなどのヘッドレスを導入すれば効率化できる
- **ユニットテスト** — テスト未実装。Vitestでコンポーネントテストを追加すれば、今後の変更が安全に
- **SEO改善** — メタタグ、Open Graphデータ、構造化データの追加で検索表示を強化
- **モーション軽減対応** — `prefers-reduced-motion` を尊重し、アニメーションに敏感なユーザーへ配慮
- **ブログセクション** — 学んだことを書くスペースを追加し、継続的な成長を発信

npm install
npm run dev
